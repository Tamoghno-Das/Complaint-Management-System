# Complaint Management System (Java + Oracle)

This project is a **simple complaint tracking app**.
Think of it like a digital register where a team can:
- receive a complaint,
- keep it in a database,
- update its status,
- and remove it if needed.

It includes **two ways to use the app**:
1. **Console mode** (text menu in terminal)
2. **Dashboard mode** (desktop window built with Swing)

---

## 1) Project in Plain Language (Non-Technical)

Imagine a customer says: _"My internet is slow."_

This app helps you do the following:
1. Save that complaint with customer name + issue details.
2. See all saved complaints.
3. Change complaint status (for example: `OPEN` -> `CLOSED`).
4. Delete a complaint if it was added by mistake.

So, in simple terms: **this app is a small complaint notebook connected to a database**.

## 8) Database Setup (SQL Scripts)

From the code, this project expects:
- table name: `complaints`
- columns: `id`, `name`, `description`, `status`
- stored procedure: `update_complaint_status`

Use the following SQL as a starter setup:

```sql
-- 1) Table
CREATE TABLE complaints (
    id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    name VARCHAR2(100) NOT NULL,
    description VARCHAR2(500) NOT NULL,
    status VARCHAR2(30) NOT NULL
);

-- 2) Stored procedure used by updateStatusUsingProcedure(...)
CREATE OR REPLACE PROCEDURE update_complaint_status (
    p_id IN NUMBER,
    p_status IN VARCHAR2
) AS
BEGIN
    UPDATE complaints
    SET status = p_status
    WHERE id = p_id;
END;
/
```

> Note: The exact SQL syntax can vary by Oracle version and identity settings.
> If identity is not supported in your setup, use a sequence + trigger approach.

## 11) Important Behavior to Know

- Dashboard calls `loadDummyData()` on startup.
- That means each app start inserts sample complaints (`Sony`, `Amit`) again.
- If you restart repeatedly, duplicate sample records can appear.

If this is not desired, remove or guard the dummy loader in `ComplaintDashboard` and `Main`.

## 12) Key Code Snippets (With Meaning)

### A) Complaint model (`Complaint.java`)

```java
@AllArgsConstructor
@NoArgsConstructor
@Getter
@Setter
@ToString
@Builder
public class Complaint implements Serializable, Cloneable {
    private Integer id;
    private String name;
    private String description;
    private String status;
}
```

What this means:
- It defines what a complaint looks like.
- Lombok generates constructors/getters/setters automatically.

### B) Add complaint (service layer)

```java
String sql = "insert into complaints(NAME,DESCRIPTION,STATUS) values(?,?,?)";
try (PreparedStatement ps = con.prepareStatement(sql)) {
    ps.setString(1, complaint.getName());
    ps.setString(2, complaint.getDescription());
    ps.setString(3, complaint.getStatus());
    ps.executeUpdate();
}
```

Meaning:
- Saves complaint data into Oracle safely using prepared statements.

### C) Update status via stored procedure

```java
try (CallableStatement cs = con.prepareCall("{call update_complaint_status(?,?)}")) {
    cs.setInt(1, id);
    cs.setString(2, status);
    cs.execute();
}
```

Meaning:
- Calls DB procedure `update_complaint_status` instead of direct SQL update.

### D) Dashboard button action (Add)

```java
addBtn.addActionListener(e -> {
    String name = nameField.getText();
    String desc = descField.getText();
    String status = statusField.getText();
    service.addComplaint(new Complaint(name, desc, status));
    showMessage("Complaint Added");
});
```

Meaning:
- When user clicks **Add**, form data is saved immediately.

## 14) Troubleshooting Guide

### Problem: "Connection refused" / cannot connect to DB
Possible reasons:
- Oracle not running
- wrong host/port/service in URL
- firewall/network issue

Fix:
- verify Oracle listener/service
- verify URL format
- verify credentials

### Problem: "table or view does not exist"
Fix:
- run the table creation SQL script
- ensure you are connected to the same schema/user

### Problem: "procedure does not exist"
Fix:
- create `update_complaint_status` procedure in same schema

### Problem: Maven build fails
Fix:
- check Java and Maven versions
- run clean build:

```powershell
mvn clean compile
```

### Problem: Dashboard opens but no data shown
Fix:
- click **View** after adding records
- verify DB writes are succeeding

## 15) Security and Production Notes

Current code uses hardcoded DB credentials in `DBUtil.java`.
For real-world projects, move credentials to:
- environment variables,
- encrypted vault,
- or secure configuration files.

Also consider:
- input validation for name/description/status
- status enum (`OPEN`, `IN_PROGRESS`, `CLOSED`)
- logging framework (instead of `System.out.println`)
- pagination for large complaint lists
